# WordPress Starter — Design Spec

**Status:** Design approved, ready for planning
**Date:** 2026-05-11
**Author:** Jonas Bergert (with brainstorming partner)

## Goal

A WordPress-native counterpart to the Payload Starter, so clients can choose between Payload (typed, code-first, Next.js) and WordPress (familiar, plugin-rich, traditional PHP hosting) while getting comparable capability from us: forkable starter, AI page composition, design tokens, block-based pages, editor-friendly admin.

The two starters are siblings — not a shared codebase — but the design philosophies match: forkable per-client, starter/client hard line, AI-first content composition, design-tokens-everywhere, clear scope boundaries.

## Non-goals (v1)

- Multilingual / i18n (deferred — Polylang/WPML integration is its own spec)
- SEO meta box (lean on Rank Math/Yoast)
- E-commerce, multisite, custom post types beyond Page/Post
- Headless WordPress (this is WP-native; clients who want headless can use the Payload starter)
- Live SSE streaming (using polling for portability)
- Block-tree diff UI for the edit flow (replace + undo in v1)
- Bundled SMTP / transactional email (README recommends WP Mail SMTP / FluentSMTP)
- Bundled cookie consent (defer to a plugin)

## Architecture

Three repos, distributed via Composer.

```

  theme.json               ← design token defaults (palette, typography, spacing, layout)
  templates/               ← FSE block templates (front-page, single, archive, page, 404)
  parts/                   ← header/footer template parts
  blocks/                  ← starter Gutenberg blocks (starter/* namespace)
    hero/
      block.json
      render.php
      edit.tsx
      style.css

    blog-index/, contact-form/

  style.css                ← theme metadata
```

### Distribution & updates

- Parent theme and AI plugin: tagged Git releases on a private GitHub org, exposed as Composer packages (via a private Composer repository URL or Satis/Packeton).
- Client onboarding: clone the client template, `composer install`, set `.env`, deploy.
- Starter improvements reach clients via `composer update bergert/wp-starter-theme` and/or `composer update bergert/wp-starter-ai`. Version-pinned, atomic, no merge conflicts.

### Versioning policy

**Strict semver.** Clients pin to `^1.0` (or whatever major they started on). Breaking changes (block attribute rename, REST shape change, theme.json schema change that doesn't merge cleanly) bump major. A changelog file (`CHANGELOG.md`) lives in each repo with migration notes per major version.

### Mapping to the Payload starter's concepts

| Payload | WordPress equivalent |
|---|---|
| `src/blocks/` (starter-owned) | `wp-starter-theme/blocks/` (namespace `starter/*`) |
| `src/blocks/client/` | `client-theme/blocks/` (namespace `client/*`) |
| `src/theme/tokens.ts` | `wp-starter-theme/theme.json` |
| `src/theme/tokens.client.ts` | `client-theme/theme.json` |
| Brand/Header/Footer globals | theme.json + FSE template parts + native Settings API Brand page |
| AI composer endpoint | `wp-starter-ai/` REST routes |
| `pnpm seed` | `wp starter-theme seed` (WP-CLI command) |
| `git merge upstream/main` | `composer update bergert/wp-starter-theme` |

## Blocks

### Catalog (v1)

All under `starter/*` namespace.

| Block | Notes |
|---|---|
| `starter/hero` | Variants via `variant` attribute: `default \| split \| centered \| media-bg`. Attributes: `headline`, `subheadline`, `ctaText`, `ctaUrl`, `mediaId`, `variant`. |
| `starter/cta` | Title, body, primary button, optional secondary button. |
| `starter/faq` | Container with inner blocks. Allowed child: `starter/faq-item`. |
| `starter/faq-item` | Question + answer (rich text). Accordion behavior on render. |
| `starter/prose` | Long-form rich text with constrained content width and typographic defaults. |
| `starter/pull-quote` | Quote + optional citation. |
| `starter/image-caption` | Image + caption + optional alt-text override. |
| `starter/stat` | Big number + label + optional context line. |
| `starter/blog-index` | Server-rendered list of latest posts with category filter attribute. |
| `starter/contact-form` | Configurable fields (name, email, optional phone, message). Recipient defaults to Brand Settings; overridable per-instance. Honeypot + 5s time-trap. |

### Registration pattern

Each block lives in `blocks/<name>/` with three files:

1. `block.json` — canonical metadata (name, title, attributes, supports, description).
2. `render.php` — server-side render callback. Receives `$attributes`, `$content`, `$block`. Outputs HTML using CSS custom properties from theme.json (never hex literals).
3. `edit.tsx` — editor UI (React, compiled via `@wordpress/scripts`).

`inc/register-blocks.php` iterates `blocks/*/block.json` and calls `register_block_type()`. Adding a block = drop a folder; no central registry edit needed. The convention is enforced by a lint rule that fails CI if a block folder lacks any of the three files.

### Client blocks

Child themes follow the same pattern in `client-theme/blocks/`. Namespace `client/*`. The child theme's `functions.php` runs the same auto-discovery iterator over its own `blocks/` directory.

For AI compatibility, client blocks **must** include in their `block.json`:

- `description` (used in the AI's tool schema)
- Explicit `attributes` (no relying on implicit attributes from `supports`)

The starter docs spell this out as the "AI-aware block" convention.

### Theme tokens (theme.json)

Parent theme `theme.json` defines:

- `settings.color.palette` — semantic tokens: `primary`, `surface`, `surface-elevated`, `text`, `text-muted`, `accent`, `border`
- `settings.typography.fontFamilies` — `body`, `heading`, `mono`
- `settings.typography.fontSizes` — fluid type scale with clamp() values
- `settings.spacing.spacingScale` — t-shirt sizes
- `settings.layout.contentSize` / `wideSize` — container widths
- `styles.blocks.starter/*` — per-block default styles using `var(--wp--preset--color--primary)` etc.

Child theme `theme.json` overrides brand-specific values (palette, font families). WP merges parent + child automatically and generates CSS custom properties. Block templates and `render.php` reference `var(--wp--preset--*)` only.

A PHP lint rule (`phpcs` custom sniff) rejects hex/rgb/hsl literals inside `blocks/**/*.{php,css}`.

### Block patterns

The parent theme ships block patterns for common compositions (hero + cta + faq, prose article, contact page layout). These appear in the inserter and are usable as starting points for human authors and as few-shot examples for the AI.

## AI plugin

### Three flows

#### A. Compose (new page from prompt)

1. User opens a page (new or existing). Document sidebar shows "AI" panel.
2. **Compose** button → modal with: prompt textarea, page-type selector (Landing / About / Services / Contact / Other), optional tone descriptor.
3. Submit → POST `/wp-json/starter-ai/v1/compose` → enqueues an Action Scheduler job, returns `{ job_id }` immediately.
4. Worker process:
   - Builds system prompt: rules + merged block schema + Brand Settings values + few-shot examples per page type + available block patterns.
   - Calls Anthropic Messages API with `web_fetch_20250910` server tool enabled + custom `emit_page(blocks: BlockTree)` tool whose input JSON schema is the merged block schema.
   - On response: validates against JSON Schema; on validation failure, one repair attempt (send errors back to Claude).
   - On success: writes result + fetched-URL list to job row, marks `complete`.
   - On error (timeout, rate limit, malformed response): marks `error` with a user-facing message.
5. Editor polls GET `/v1/jobs/{job_id}` (shared status endpoint for compose + edit) every 750ms. Response shape:
   ```ts
   {
     status: 'queued' | 'fetching' | 'composing' | 'complete' | 'error',
     urls_fetched: string[],
     progress_note?: string,
     result?: BlockTree,
     error?: string
   }
   ```
6. Pills appear as `urls_fetched` grows. Spinner during `composing`.
7. On `complete`: editor calls `wp.blocks.parse()` on serialized markup → inserts blocks at cursor or replaces post content (user picks in modal).

#### B. Edit (modify existing page)

1. Document sidebar **Edit with AI** button → modal with prompt input. Placeholder: "Add an FAQ section, shorten the hero, change CTA to…"
2. Modal includes warning: "This will replace your current page. Use Undo to revert."
3. Plugin parses `post_content` via `Parser.php` → JSON block tree.
4. POST `/wp-json/starter-ai/v1/edit` with `{ tree, instruction }` → enqueues a job. Same polling pattern as compose.
5. Worker builds prompt with existing tree + instruction + schema + brand context. web_fetch enabled.
6. Result replaces post content via `dispatch('core/block-editor').resetBlocks()`. Native undo works as expected.

No diff UI in v1. Deferred to v2 if editor feedback demands it.

#### C. Refine (single block)

1. User selects any block. Block-level AI panel shows in the sidebar.
2. Quick actions (contextual to block type): for CTA → "Make more urgent", "Shorter"; for Hero → "Try a different angle", "More benefit-led"; etc.
3. Custom instruction input always available.
4. Action → POST `/wp-json/starter-ai/v1/refine` (blocking, no polling) with `{ blockName, attributes, innerBlocks, instruction }`.
5. Backend uses single-block schema + current attributes + instruction. `emit_block(attributes, innerBlocks)` tool.
6. Returns new attributes. Editor merges via `updateBlockAttributes` + `replaceInnerBlocks`.

web_fetch is **off by default** for refine. Refines are short and should stay focused.

### Block-tree schema

Built at **runtime**, not release time. The starter theme and AI plugin ship as independent Composer packages with independent release cycles — a build-time schema would couple them. Instead:

- `SchemaBuilder` queries `WP_Block_Type_Registry::get_instance()->get_all_registered()` per request.
- Filters to: all `starter/*` blocks, all `client/*` blocks, plus a curated allowlist of core blocks needed for prose content (`core/paragraph`, `core/heading`, `core/list`, `core/list-item`, `core/image`, `core/separator`). Core block schemas are hard-coded in the plugin since they're stable WP APIs.
- For each block: reads `attributes`, `description`, `parent`, `ancestor`, `supports` from the registered type.
- Emits the final Anthropic tool input JSON schema for `emit_page(blocks: BlockTree)`.

Result shape per block:

```jsonc
{
  "blocks": {
    "starter/hero": {
      "description": "...",
      "attributes": { "variant": { "type": "string", "enum": [...] }, ... },
      "allowsInnerBlocks": false
    },
    "starter/faq": {
      "description": "...",
      "attributes": { ... },
      "allowsInnerBlocks": true,
      "allowedChildBlocks": ["starter/faq-item"]
    }
  }
}
```

Result is cached in a transient (1 hour TTL, invalidated when `WP_Block_Type_Registry` changes — hooked via `register_block_type` action) so SchemaBuilder doesn't re-run on every API call.

Client blocks that don't follow the AI-aware convention (missing `description` or implicit attributes via `supports` only) are excluded from the schema with a warning surfaced on the AI plugin's settings page.

An optional `wp-starter-ai/schema/blocks.json` cache file may be committed for offline testing — populated by a `wp starter-ai dump-schema` WP-CLI command that introspects a running WP instance. Not used at runtime.

### Anthropic API integration

- No official PHP SDK; thin custom client wraps the Messages API via `wp_remote_post()`.
- **Models** (configurable per flow):
  - Compose: `claude-sonnet-4-6` (default)
  - Edit: `claude-sonnet-4-6` (default)
  - Refine: `claude-haiku-4-5` (default — much cheaper, fast enough for single-block work)
- **Prompt caching** on the system prompt (block schema + brand context). Same brand/schema across calls → significant cost reduction.
- **Timeouts**: 90s for compose/edit jobs (background, doesn't block HTTP request), 15s for refine.
- **Retries**: one retry on 429/5xx with exponential backoff. No retry on 4xx other than 429.

### Config

API key resolution order:

1. `ANTHROPIC_API_KEY` constant (set in Bedrock `.env`) — preferred for production.
2. AI plugin settings page (stored encrypted in `wp_options` using a key derived from `wp_salt()`).
3. If neither: AI panel shows a "Set up AI" CTA pointing to the settings page.

Settings page also exposes: per-flow model choice, rate limits, usage rollup, mock-mode toggle for dev.

### Rate limiting

Per-user transients. Defaults:

- 30 compose calls/hour
- 30 edit calls/hour
- 200 refine calls/hour

Configurable in settings. Prevents accidental budget burn from a single editor.

### Usage telemetry

`wp_starter_ai_usage` table: `id, user_id, kind, model, input_tokens, output_tokens, cache_read_tokens, cache_write_tokens, web_fetch_count, cost_estimate_usd, created_at`.

Cost estimate computed from current published Anthropic prices, configurable in plugin source for when prices change. Settings page shows month-to-date totals.

No data leaves the site.

### Mock mode

When `STARTER_AI_MOCK=true` in env (or in plugin settings for quick toggle), `MockProvider` returns fixture responses:

- Hand-crafted JSON files in `src/Mock/fixtures/`
- One fixture per page type for compose: `landing.json`, `about.json`, `services.json`, `contact.json`
- Fixtures for refine keyed by block name: `refine-hero.json`, `refine-cta.json`, etc.
- Fixtures for edit: `edit-add-faq.json`, `edit-shorten.json`
- Tests run in mock mode by default. A manual `scripts/test-live-compose.sh` runs against the real API as a smoke check.

### Privacy / GDPR

The plugin sends to Anthropic: the user's prompt, Brand Settings values (name, tagline, tone), and for the Edit flow the current page's block tree. web_fetch can fetch publicly accessible URLs via Anthropic's infrastructure.

README ships disclosure boilerplate for client privacy policies covering: AI processing in the editor (server: Anthropic, US-based by default — note EU region if needed), data sent, web fetching behavior.

## Brand Settings

Native WP Settings API admin page registered in `wp-starter-theme/inc/brand-settings.php`. Fields:

- Brand name, tagline, voice/tone descriptor (text)
- Logo (image, `wp.media` picker)
- Default OG image (image)
- Primary contact email (used as contact-form recipient default)
- Social links (hand-rolled repeater: platform + URL)
- Phone, address (text, optional)

Values stored in `wp_options` under a single serialized array (`starter_theme_brand`). Accessor: `Starter\Brand::get('contact_email')`.

The Brand page is registered by the parent theme, not a plugin — the theme is the product, and clients aren't switching themes. Values persist in the DB regardless if a switch ever happens.

## Contact form

`starter/contact-form` block configuration via block attributes:

- Fields shown: `name` (required), `email` (required), `phone` (optional), `message` (required)
- Recipient: defaults to Brand Settings primary contact email; overridable per-instance
- Success message (rich text)
- Redirect URL (optional)

Spam protection:

- Honeypot field (CSS-hidden, must remain empty)
- 5-second time-trap (submission rejected if form is submitted in under 5s)
- No third-party captcha dependency in v1

Submission flow:

1. JS submits to REST endpoint `/wp-json/starter/v1/contact`
2. Server validates honeypot + time-trap + required fields + email format
3. Stores submission as a `contact_submission` CPT (private, admin-only)
4. Sends notification email via `wp_mail()` to recipient
5. Returns `{ ok: true, message }` → block JS shows success state inline

`contact_submission` CPT:

- `public => false`, `show_ui => true`, `capability_type => 'manage_options'`
- Columns: date, name, email, excerpt of message
- Read-only in admin (no edit screen)
- 90-day retention; daily WP-cron task deletes older entries

The README's deployment notes recommend WP Mail SMTP or FluentSMTP for actual deliverability. Native `wp_mail()` is unreliable on most hosts.

## Site identity

- **Header** — FSE template part `parts/header.html`. Logo (from Brand Settings) + site title + navigation block. Editable in Site Editor.
- **Footer** — FSE template part `parts/footer.html`. Brand info + social links + secondary nav. Editable in Site Editor.
- **Templates** — `templates/index.html`, `single.html`, `page.html`, `archive.html`, `front-page.html`, `404.html`. All standard FSE block templates referencing the header/footer parts.

## Pages versioning

Use WP's built-in post revisions. No additional work. Editors get revision history per page out of the box.

## Testing

- **PHP unit tests** (PHPUnit via `wp-env run tests-wordpress`):
  - Block render functions (each `render.php` with attribute fixtures)
  - Contact form submission (validation, honeypot, CPT creation, mail)
  - Anthropic `Client` with HTTP mocks
  - `BlockTree\Serializer` and `Parser` round-trip tests
  - `SchemaBuilder` merging starter + client blocks
  - `RateLimiter` and `Tracker` behavior
- **Editor JS tests** (Jest via `@wordpress/scripts`):
  - Polling state machine
  - Attribute merge logic for refine
  - Sidebar UI components
- **E2E** (Playwright targeting `wp-env`):
  - Front-page loads, header/footer render
  - Open editor, drag a Hero block, set attributes, publish, verify rendered HTML
  - AI Compose in mock mode → blocks appear in editor → publish → verify rendered HTML
  - AI Refine in mock mode → block attributes change
  - AI Edit in mock mode → page content changes
  - Contact form submission → CPT row created → email triggered (mock mailer)
- **Live API smoke test**: `scripts/test-live-compose.sh` runs one real compose against Anthropic. Run manually before release, not in CI.

CI runs PHP + JS unit tests + Playwright on every PR. Linting includes:

- `phpcs` with custom sniff rejecting hex/rgb/hsl literals in `blocks/`
- ESLint on editor JS
- Custom lint rule failing CI if any `blocks/<name>/` is missing `block.json`, `render.php`, or `edit.tsx`

## Local development

### Starter author (you)

- `wp-env start` in either `wp-starter-theme/` or `wp-starter-ai/`. Mounts the package into a fresh WP install with MariaDB.
- `pnpm dev` runs `@wordpress/scripts start` for editor JS hot-reload.
- For working on both simultaneously, the AI plugin's `.wp-env.json` declares the parent theme as a `mappings` entry pointing at the local theme repo.

### Client developer

- Works in the Bedrock client repo.
- `composer install` pulls starter theme + AI plugin at pinned versions.
- `wp-env start` (recommended for parity) or any local WP pointed at `web/wp/`.
- Edits only `web/app/themes/client-theme/`.

## Deployment notes (for client repos)

Targeted at clients deploying to Hetzner Webhosting, Hetzner Cloud, or comparable European hosts. README will include host-specific notes:

- **Hetzner Cloud / VPS / dedicated**: full control. Bedrock standard install. nginx + PHP-FPM with `max_execution_time` ≥ 120s. No SSE config needed since we use polling.
- **Hetzner Webhosting (managed shared)**: Bedrock works, follow their non-standard webroot convention; Composer-install via SSH or pre-deploy CI step. PHP execution time configurable via `.htaccess`. Action Scheduler needs WP cron working — verify cron is enabled.
- **Cloudflare in front**: bypass for `/wp-json/starter-ai/*` if a client adds Cloudflare proxy mode. Standard recommendation, not required.

## Open questions punted to planning

These don't block design approval but need answers during implementation planning:

1. **Composer distribution**: GitHub private composer-installer, Satis self-hosted, or Packeton? Decision affects per-client setup complexity. Default plan: GitHub private repos with `composer/installers` + a private packages.json — simplest path.
2. **Action Scheduler vs alternatives**: confirm Action Scheduler is the right choice vs a bespoke transient + WP-cron pattern. Action Scheduler adds a dependency; bespoke adds code. Default plan: Action Scheduler — battle-tested.
3. **Editor sidebar UX details**: exact placement of Compose / Edit buttons, modal layouts, pill component design. Planning phase will produce mockups.
4. **Few-shot examples**: how many per page type, who writes them, how often they're refreshed. Default plan: 2–3 per page type, written by you during implementation, versioned with the plugin.
5. **CI/CD**: how releases are tagged, how Composer sees them, how clients are notified of updates. Default plan: GitHub Actions release workflow, semantic-release-style.

## Out of scope (explicitly)

- Bundling translations / i18n / multilingual support
- Bundling SEO meta box, sitemap generator, schema markup
- Bundling cookie consent banner
- Bundling SMTP / email delivery
- E-commerce, multisite, headless mode
- Block-tree diff UI for the Edit flow
- Live SSE streaming
- More than the 10 v1 blocks
- Mobile editor support (beyond what WP core provides)
- Custom field types beyond what Gutenberg / theme.json offers
